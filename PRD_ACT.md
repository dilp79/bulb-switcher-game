# Канонические требования проекта Bulb Switcher

## 1. Продукт

**Bulb Switcher** — статическая браузерная головоломка, в которой игрок нажимает на кнопки и переводит все лампочки в состояние `выключено`.

Ключевая идея:
- связи между кнопками и лампочками неочевидны;
- одно нажатие меняет состояние всех связанных лампочек;
- лампочки проходят циклические состояния `off -> green -> yellow -> red -> ... -> off`;
- победа засчитывается только когда все лампочки выключены.

## 2. Канонический scope

### 2.1 Игровые экраны

Приложение должно содержать:
- главный экран;
- экран выбора уровней;
- экран игры;
- экран справки;
- экран редактора уровней.

### 2.2 Игровая механика

Обязательные правила:
- у уровня есть `buttons_count`, `bulbs_count`, `colors_count`, `connections`, `initial_states`, `target_states`;
- `target_states` по умолчанию равен массиву из нулей;
- нажатие на кнопку увеличивает состояние каждой связанной лампочки на `1` по модулю `colors_count + 1`;
- победа наступает только если все текущие состояния совпадают с `target_states`;
- для уровня считаются ходы и время;
- у игрока есть одна подсказка на уровень;
- подсказка работает в два шага: включение режима подсказки, затем выбор кнопки;
- после выбора подсказка на 3 секунды показывает связи выбранной кнопки с лампочками и больше в этом уровне недоступна;
- при нажатии кнопки проигрывается `assets/sounds/button_click.mp3`;
- при использовании подсказки проигрывается `assets/sounds/hint.mp3`;
- при победе проигрывается `assets/sounds/victory.mp3` и показывается салют на базе `assets/images/confetti.gif`.

### 2.3 Каталог уровней

Игра должна поддерживать:
- встроенные уровни из репозитория;
- пользовательские уровни, сохранённые локально в браузере.

Правило согласования конфликтов в старых документах:
- старый PRD описывает 3 блока;
- фактический репозиторий содержит 5 встроенных блоков;
- новая реализация обязана поддерживать **все встроенные блоки, реально находящиеся в репозитории**, а не ломаться из-за жёсткой привязки к первым трём.

### 2.4 Выбор уровней

Экран выбора уровней должен:
- показывать все встроенные блоки;
- показывать блок пользовательских уровней;
- отображать состояние прохождения;
- отображать лучший результат по времени и ходам;
- позволять открыть любой уровень напрямую.

### 2.5 Прогресс

В браузере без авторизации сохраняются:
- последний открытый уровень;
- факт прохождения уровня;
- лучший результат по времени и ходам;
- пользовательские уровни;
- пользовательские настройки интерфейса, если они будут добавлены.

Источник хранения:
- `localStorage`.

Важно:
- браузерная версия не может писать пользовательские уровни обратно в файловую систему репозитория;
- единственным рабочим хранилищем пользовательских уровней считается `localStorage`.

### 2.6 Редактор уровней

Редактор обязан поддерживать:
- создание нового уровня;
- редактирование сохранённого пользовательского уровня;
- изменение названия уровня;
- изменение числа кнопок от `1` до `8`;
- изменение числа лампочек от `1` до `8`;
- изменение числа цветов от `2` до `4`;
- генерацию нового набора связей;
- ручное переключение связи между конкретной кнопкой и лампочкой;
- генерацию начального состояния;
- ручное изменение стартового состояния лампочек;
- расчёт сложности по формуле;
- валидацию уровня перед сохранением;
- удаление пользовательского уровня;
- немедленный переход к игре на сохранённом уровне.

Правила валидации:
- каждая кнопка должна иметь минимум одну связь;
- каждая лампочка должна иметь минимум одну связь;
- стартовое состояние должно содержать хотя бы одну включённую лампочку.

### 2.7 Генерация уровней

Генератор обязан:
- работать в диапазонах `buttons_count: 1..8`, `bulbs_count: 1..8`, `colors_count: 2..4`;
- создавать такие связи, чтобы каждая кнопка и каждая лампочка имели минимум одну связь;
- рассчитывать сложность по формуле:

```text
complexity = 1 + log10((buttons_count + bulbs_count) * connections_count)
```

- генерировать стартовое состояние из решаемой конфигурации, а не случайным шумом;
- гарантировать, что хотя бы одна лампочка включена.

## 3. UX-требования

Интерфейс должен:
- нормально работать на desktop, tablet и mobile;
- не зависеть от фиксированного canvas с жёсткими координатами;
- одинаково поддерживать мышь и touch;
- явно показывать состояние игры: активный уровень, ходы, таймер, рекорд, доступность подсказки;
- после победы предлагать переход к следующему уровню;
- позволять вернуться к выбору уровней и в главное меню без потери сохранённого прогресса.

## 4. Технические требования

### 4.1 Платформа

Целевая поставка:
- статический веб-проект для GitHub Pages;
- запуск без сервера приложений;
- данные хранятся локально в браузере.

### 4.2 Активы

Используются только существующие ассеты:
- `assets/images/bulb_off.png` — 512x512;
- `assets/images/bulb_green.png` — 512x512;
- `assets/images/bulb_yellow.png` — 512x512;
- `assets/images/bulb_red.png` — 512x512;
- `assets/images/button.png` — 128x128;
- `assets/images/confetti.gif` — 300x300;
- `assets/sounds/button_click.mp3`;
- `assets/sounds/hint.mp3`;
- `assets/sounds/error.mp3`;
- `assets/sounds/victory.mp3`.

Новые изображения и звуки без согласования не добавляются.

### 4.3 Визуальная система

Новая реализация должна:
- поддерживать единый visual language между главной, игрой, выбором уровней и редактором;
- ощущаться как puzzle console, а не как generic dashboard;
- использовать сильную типографическую иерархию и явные игровые акценты;
- сохранять читаемость на mobile без потери атмосферы.

### 4.4 Архитектурные требования

Новая реализация должна:
- иметь единый источник истины для каталога уровней, пользовательских уровней и прогресса;
- не опираться на глобальные мутабельные объекты сцен;
- отделять игровую логику от отрисовки;
- позволять расширять число встроенных блоков без переписывания UI;
- быть читаемой и поддерживаемой без Phaser-specific кода, если Phaser не даёт продуктовой пользы.

## 5. Definition of Done

Задача считается выполненной, когда:
- требования сведены в единый документ;
- приложение переписано на новую архитектуру;
- встроенные блоки загружаются и отображаются корректно;
- игра, выбор уровня, справка и редактор работают end-to-end;
- прогресс и пользовательские уровни сохраняются в `localStorage`;
- документация описывает уже новую реализацию, а не старую.
